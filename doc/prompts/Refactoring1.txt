====================================================

Apply a standardized exception strategy:

1. Introduce domain-specific exceptions where needed
(e.g., `OrderNotFoundException`, `InvalidTokenException`)
2. Do NOT catch generic `Exception`
3. Do NOT swallow exceptions
4. Use a centralized `@ControllerAdvice`
(e.g., `GlobalExceptionHandler`)

Global Exception Handler Responsibilities:
- Map exceptions to correct HTTP status codes
- Return consistent error response structure
- Log exceptions appropriately (without leaking sensitive data)

Exception Context:

When throwing exceptions, the message MUST include the relevant prohibited value or ID (e.g., throw new UserAlreadyExistsException("Email already exists: " + email)).
Do not throw generic RuntimeException or ResponseStatusException in Service layers; use the custom exceptions defined in com.example.ordermgmt.exception (e.g., ResourceNotFoundException, InvalidOperationException).

- You may also go with industry standard way.

Apply standard logging practices:

✔ Use SLF4J (`LoggerFactory`)
✔ No `System.out.println`
✔ No excessive logging
✔ No logging of sensitive data (passwords, tokens, secrets)

Logging Guidelines:
Event -> Action -> Result format:

Strict Logging Format (Event -> Action -> Result): Eliminate noise. Use exactly this pattern for every Service method:

Entry (Event): logger.info("Processing [MethodName] for [Entity]: {}", id);
Guard Clauses/Decisions (Action): logger.warn("Skipping [MethodName] for [Entity]: {} - [Reason]", id);
Success (Result): logger.info("[MethodName] completed successfully for [Entity]: {}", id);
Error (Result): logger.error("[MethodName] failed for [Entity]: {}: {}", id, ex.getMessage());

Logs must be:
- Searchable
- Consistent
- Meaningful
- Noise-free
