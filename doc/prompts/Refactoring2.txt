Refactoring Tasks

1. Logging Consistency Across All Files
Goal: Ensure every file in the entire project follows a consistent logging strategy.

Apply standard logging across controllers, services, and repositories.
Use SLF4J (`LoggerFactory`), following the pattern:

  Entry: `logger.info("Processing [MethodName] for [Entity]: {}", id);`
  Guard Clauses/Decisions: `logger.warn("Skipping [MethodName] for [Entity]: {} - [Reason]", id);`
  Success: `logger.info("[MethodName] completed successfully for [Entity]: {}", id);`
  Error: `logger.error("[MethodName] failed for [Entity]: {}: {}", id, ex.getMessage());`
Logs must be:

  Searchable
  Consistent
  Meaningful
  Noise-free
Ensure no sensitive information is logged (e.g., passwords, tokens).

2. Proper HTTP Status Codes for Each Endpoint
Goal: Ensure that each API endpoint returns the correct HTTP status code for different types of requests (GET, POST, PUT, DELETE).

Inspect all controllers and ensure they return the correct HTTP status codes based on the request.

  200 OK for successful GET/POST requests.
  201 Created for successfully created resources.
  204 No Content for successful DELETE requests.
  400 Bad Request for validation failures or incorrect requests.
  404 Not Found for resources that don't exist.
  500 Internal Server Error for server-side issues.
  And many more appropriate...
Implement these status codes across all APIs, ensuring consistency and correct status handling for success, error, and validation cases.

3. Transaction Management (Transactional Annotation)
Goal: Ensure that the `@Transactional` annotation is applied only where necessary.

Review all service methods and confirm that only methods dealing with critical business transactions have the `@Transactional` annotation.
Avoid using `@Transactional` on non-transactional methods, such as read-only service methods.
Where applicable, ensure correct propagation and isolation settings are used based on the business requirements of the method.

4. Exception Handling in Services and Global Handler
Goal: Ensure proper exception handling throughout the project. There is global exception handler use that properly.

Every service method may have appropriate exception handling for both business-level errors and unexpected errors as they needed.
Introduce domain-specific exceptions (e.g., `OrderNotFoundException`, `InvalidTokenException`, etc.), and avoid generic exceptions like `RuntimeException`.
Ensure a global exception handler (`@ControllerAdvice`) is in place to catch and map exceptions to appropriate HTTP status codes.

  Example: `throw new UserAlreadyExistsException("Email already exists: " + email);`
  Log exceptions appropriately without leaking sensitive information.
  Ensure the global handler returns a consistent error structure and maps exceptions to correct status codes (e.g., `404` for `NotFound`, `400` for `BadRequest`, `500` for `InternalServerError` etc.).
Make sure no exception is silently swallowed.



General Refactoring Approach

Refactoring Goals:

1. Logging: Ensure consistency in how logging is done throughout the project, using SLF4J and following the outlined structure.
2. HTTP Status Codes: Audit all API endpoints and correct HTTP status code handling based on request types and scenarios.
3. Transactional Integrity: Ensure that only appropriate service methods are transactional, and that transaction management settings are correctly configured.
4. Exception Handling: Introduce domain-specific exceptions, implement centralized exception handling, and ensure proper logging and HTTP status mappings.

Safety Constraints

DO NOT change:

  Existing business logic
  API request/response structure
  Database schema or entity mappings
  Security flow
  External integrations or configuration values
  Transaction boundaries
  Performance characteristics unless explicitly directed
DO NOT risk behavior changes or API changes.
Ensure the project behaves exactly the same after the refactor as before, without any unintended side effects.
